## 11.4 Sistema AI/LLM

### 11.4.1 Anatomia del cluster

Il cluster `ai/` ha undici file e tre punti d'ingresso distinti, non uno solo. Il file piГ№ centrale per coverage di funzionalitГ  ГЁ `ai/core.cljs` (370 righe), ma ГЁ "core" solo per il flusso di **codegen tier-aware**: orchestra prompt building, dispatch provider, parsing della risposta tier-1 (raw code) o tier-2/3 (JSON con `{:type :code|:clarification}`). Vedi `ai/core.cljs:322-370` per `generate`. Le altre due funzionalitГ  maggiori (describe e auto-session) hanno il loro core separato.

`ai/prompts.cljs` (995 righe) ГЁ quasi interamente contenuto: 4 grossi `def` per `tier-1-prompt`, `tier-2-prompt`, `tier-3-prompt`, `auto-refinement-prompt`, piГ№ 2 `defn` micro (`get-default-prompt`, `tier->id`, `get-prompt`) negli ultimi 25 righe. La logica ГЁ ~3% del file, il resto ГЁ la "DSL reference" inserita dentro il system prompt.

`ai/describe.cljs` (483 righe) ГЁ il modulo della feature `(describe)`: collect-metadata di una mesh, multimodal API call, provider-specific multimodal message building (`build-user-message` ha case per anthropic/openai/google/ollama in [L344-379](src/ridley/ai/describe.cljs#L344-L379)), parsing della risposta che puГІ chiedere extra views via JSON.

`ai/describe_session.cljs` (282 righe) ГЁ l'orchestratore: espone `(describe)`, `(ai-ask)`, `(end-describe)`, `(cancel-ai)`. Tiene la conversation history, la refinement-chain (max 2 round extra di richieste viste/slice), e gli AbortController per cancellazioni.

`ai/auto_session.cljs` (461 righe) implementa `/ai-auto`: genera codice tier-3 в†’ esegue в†’ cattura viste в†’ invia al modello vision в†’ applica correzioni в†’ itera. Ha la sua `call-refinement-api` ([L214-289](src/ridley/ai/auto_session.cljs#L214-L289)) duplicata da `ai/core` e `ai/describe`. Interfaccia di start/cancel/continue.

`ai/batch.cljs` (445 righe) ГЁ una test harness completamente indipendente: parsa file EDN di test suite, esegue in sequenza chiamate a `ai/generate`, cattura screenshot, esporta ZIP con prompt.txt + generated-code.clj + screenshot.png + summary.json + README.md. Anche un parser "inline" pipe-separated per `/ai-batch-inline`.

`ai/capture_directives.cljs` (163 righe) parsa direttive `[view: front]` `[slice: z=30]` `[no-default-views]` dai prompt utente, le esegue su `viewport/capture`, ritorna `[label data-url]` pairs. Usato da auto-session e describe-session.

`ai/history.cljs` (126 righe, giГ  letto in fase precedente) gestisce i blocchi commentati `;; в”Ђв”Ђ AI step N в”Ђв”Ђ` come storico in-source. Funzioni pure di formattazione testo: `step-block`, `next-step-number`, `find-all-step-ranges`, `history-for-prompt`. Chiamato da `ai/core`, da `auto-session`, e da `codemirror.cljs` (pattern documentato in fase 2 intervista 1).

`ai/prompt_store.cljs` (148 righe) ГЁ il livello di customizzazione: localStorage con cascade `id:provider:model в†’ id:provider в†’ id в†’ default`, con macro expansion `{{macro}}` в†’ context map, ed export/import JSON. Cinque ID prompt registrati ([L11-22](src/ridley/ai/prompt_store.cljs#L11-L22)): `codegen/tier1`, `codegen/tier2`, `codegen/tier3`, `describe/system`, `describe/user`.

`ai/rag.cljs` (55 righe) ГЁ una RAG keyword-based su Spec.md per tier-3. Tokenize query, score per overlap (3Г— keyword + 2Г— title + 1Г— content), ritorna top-k chunks. Solo invocato in `ai/core/generate` quando tier=:tier-3.

`ai/rag_chunks.cljs` (287 righe) ГЁ il file dei chunks. Contenuto, non logica.

In sintesi, il "core" funzionale ГЁ triplo, non singolo: `ai/core` ГЁ solo il core del codegen testuale; `ai/describe` ospita anche dispatch multimodal proprio; `ai/auto_session` costruisce sopra entrambi e ha refinement-loop suo. Il prompt store ГЁ infrastruttura cross-feature, history e RAG sono adiacenti, capture-directives ГЁ utility, batch ГЁ isolato.

### 11.4.2 I provider e l'astrazione che li unisce

**Non c'ГЁ astrazione comune. Tre `case` distinte in tre file diversi.** Vedi `ai/core.cljs:355-364` (case su `provider-name` per dispatch a `call-anthropic`, `call-openai-compatible`, `call-ollama`, `call-google`), `ai/describe.cljs:419-435` (`dispatch-call`, identico schema con varianti per multimodal), `ai/auto_session.cljs:231-289` (`call-refinement-api`, ancora schema simile con JSON mode forzato per Google). Ogni file ha le sue funzioni private `call-anthropic`, `call-openai-compatible`/`call-openai`, `call-ollama`, `call-google`, alcune con piccole differenze (es. timeout, signal opzionale). Niente protocol, niente multimethod, niente factory di provider. Aggiungere un sesto provider richiede toccare almeno tre file.

Cosa ГЁ specifico di ogni provider, riassunto. **Anthropic**: `system` come parametro top-level separato dai messages, non come ruolo. Header `x-api-key` + `anthropic-version`. Image come `{:type "image" :source {:type "base64" :media_type "image/png" :data ...}}` (vedi `describe.cljs:201-205`). Header speciale `anthropic-dangerous-direct-browser-access: true` per bypassare CORS dal browser. **OpenAI/Groq**: stessa shape (chat completions), `system` come messaggio `{:role "system"}` prepended. Auth `Authorization: Bearer`. Image come `{:type "image_url" :image_url {:url ... :detail "low"}}`. Groq punta a `api.groq.com/openai/v1/chat/completions`, ГЁ OpenAI-compatible. **Ollama**: `messages` con role/content, `system` come messaggio prepended, `stream: false` esplicito. Image come campo `:images` separato sul messaggio user (array di base64), non blocco inline. Niente API key, ГЁ locale via `ollama-url` setting. **Google Gemini**: shape completamente diversa. `contents` invece di `messages`, `parts` invece di `content`, `model` invece di `assistant` per il role. `system_instruction` separato. Supporta JSON structured output via `responseMimeType` + `responseSchema` (sfruttato in tier-2/3 per forzare formato strutturato, vedi `ai/core.cljs:271-281`). API key passata in URL query string, non header.

Streaming: nessun provider lo usa nel codice attuale. Ollama lo disabilita esplicitamente. Gli altri non lo richiedono. Le risposte arrivano in `await response.json()`.

Tool use / function calling: nessun provider lo usa. Solo testo (e immagini per vision).

Asimmetria di vision (immagini in input). Tre liste in `ai/describe.cljs:385-399`: `vision-capable-providers` whitelist `#{"anthropic" "openai" "google"}` (Groq escluso anche se ha alcuni model vision-capable), `vision-capable-models` whitelist specifica per ollama (`llava`, `llava:13b/34b`) e qualche llama vision preview. La gestione dell'asimmetria ГЁ gating upstream: `vision-capable?` в†’ throw se non capable. Auto-session se non c'ГЁ vision skippa la refinement loop e si ferma al primo round. Describe lancia errore esplicito con messaggio user-friendly (`"requires a vision-capable model. Use Gemini, Claude, or GPT-4o."`).

Structured output: solo Google ha JSON mode nativo (responseSchema). Per Anthropic/OpenAI/Ollama il tier-2/3 si affida a "JSON output" come istruzione di prompt e parsing best-effort, con fallback `extract-code-from-json` e `extract-code` (markdown fence stripping) in `ai/core.cljs:67-99`.

### 11.4.3 Il pannello prompt e il prompt store

Il pannello [`ui/prompt_panel.cljs`](src/ridley/ui/prompt_panel.cljs) (285 righe) permette di **editare i system prompt usati dal cluster AI**, non di selezionare quale prompt usare per quale feature: la mappatura featureв†’prompt-id ГЁ hardcoded (codegen в†’ tier-1/2/3 in base al tier corrente, describe в†’ describe/system + describe/user). Il pannello ha tre regioni: lista prompt a sinistra (5 voci raggruppate per category, "AI Assistant" e "Accessibility"), variant selector dropdown in alto a destra (`default | <provider> | <provider>/<model>`), e textarea editor con info row sopra che mostra le macro disponibili per quel prompt. Quattro action buttons: Save (scrive in localStorage al variant selezionato), Reset to Default (cancella la chiave custom), Export All (JSON download), Import (JSON upload). Un badge `в—Џ` modified compare accanto al nome del prompt se esiste un override custom in qualche variant.

Lo store dei prompt vive in **localStorage**, non in `settings`. La cascade ГЁ in `ai/prompt_store.cljs:81-94`: `provider+model` в†’ `provider` в†’ `id senza variant` в†’ `default-text`. Il default ГЁ hardcoded in `ai/prompts.cljs` per i tre tier codegen e in `ai/describe.cljs:88-138` per `default-system-prompt` e `default-user-prompt` di describe. Quindi per ogni prompt ci sono fino a tre livelli di override: globale, per-provider, per-provider+model. Il default in codice non si tocca, l'utente puГІ solo aggiungere override sopra di esso. Le modifiche sopravvivono alla sessione perchГ© localStorage persiste, e sono esportabili come JSON per migrazione fra browser.

Macro expansion: in `prompt_store.cljs:100-112`, regex `\{\{([^}]+)\}\}` su template, lookup in context map. Macro previste: `{{source-code}}`, `{{history}}`, `{{query}}` per codegen, `{{metadata}}` e `{{object-bounds}}` per describe. Una macro non risolta lascia il placeholder e logga warning.

Una stranezza: `ai/core.cljs/build-user-content` ([L154-170](src/ridley/ai/core.cljs#L154-L170)) costruisce `<history>`, `<script>`, `<reference>` blocchi e li concatena con `\n\n` prima di passarli come prompt utente. NON usa `expand-macros` del prompt-store. Il sistema di macro ГЁ quindi parzialmente implementato: lo usa solo `ai/describe` (per `{{metadata}}`, `{{object-bounds}}`), non `ai/core`. Il codegen non sostituisce macro nel system prompt ma costruisce strutture XML-style nel user content.

### 11.4.4 Settings e capture come moduli adiacenti

**`settings.cljs` confermato.** Lo stato ГЁ esattamente quello descritto in fase 1: `:ai {:enabled, :provider, :anthropic-key, :openai-key, :groq-key, :google-key, :ollama-url, :ollama-model, :model, :groq-model, :google-model, :tier}` + `:audio-feedback`. L'unica chiave non-AI ГЁ `audio-feedback`. La UI di settings vive in [`core.cljs:1281-1605`](src/ridley/core.cljs#L1281), e gestisce nello stesso pannello tutte le chiavi: dropdown provider (con ramo on-change che reset connection-status), input API key per ogni provider, dropdown model con detect-tier che mostra il tier risultante, dropdown override tier (auto/1/2/3), test connection button, ollama URL + connection check, ollama model, e in fondo una checkbox isolata "Audio feedback on eval". Il pannello ГЁ dominato visivamente dall'AI, l'audio ГЁ un'aggiunta in coda. La promessa di "settings generici" non ГЁ onorata: 95% delle chiavi e dell'UI sono AI.

**`viewport/capture.cljs`** (592 righe) ha API piГ№ ampie delle sei viste ortografiche. Le funzioni principali, in ordine di rilevanza per i chiamanti: `render-view <spec>` (spec puГІ essere keyword `:front/:back/:left/:right/:top/:bottom/:perspective` oppure vector di direzione `[x y z]` per vista custom), `render-all-views` (tutte e 7), `render-slice <target> <axis> <position>` (sezione trasversale di una mesh specifica a una coordinata), capture screenshot blob (per export PNG e per batch), gestione zip multi-immagine via JSZip. Il modulo ГЁ in `viewport/` ma ГЁ una superficie generica: rende offscreen tramite `WebGLRenderTarget` sul renderer Three.js esistente, con bounding-box auto-fit, palette deterministica per oggetti registrati, e zip combiner. Non ha API esplicitamente "produce input per vision", ma `render-slice` esiste solo per il workflow di vision (l'occhio umano del viewport non guarda mai una sezione 2D).

I quattro chiamanti AI usano subset diversi e per ragioni diverse. **`ai/auto_session/capture-default-views`** ([L100-122](src/ridley/ai/auto_session.cljs#L100-L122)) usa solo 3+1: perspective + front + top + slice-z a metГ  altezza della prima mesh visibile, a 256Г—256. Le viste sono mirate al loop iterativo: poche immagini per ogni round per non saturare il context window e velocizzare. **`ai/describe-session/render-standard-views`** ([L47-51](src/ridley/ai/describe_session.cljs#L47-L51)) usa `render-all-views` (tutte e 7), perchГ© ГЁ una conversazione approfondita con l'utente che vede una sola volta la geometria iniziale. Plus extra dinamici quando l'AI risponde con `{"need_more": true, "requests": [...]}`. **`ai/capture-directives/execute`** ГЁ invocato sia da auto-session che da describe-session: esegue solo le viste/slice esplicitamente richieste dall'utente nel prompt come `[view: ...]` o `[slice: z=30]`. **`manual/export`** usa solo `render-view :perspective` a 600Г—400 per le immagini del manuale, niente slice.

### Sintesi 11.4

Il cluster AI **tocca tutto perchГ© lo deve fare per natura**. Vision richiede capture, codegen multi-turno richiede history visibile nel sorgente (gli `;; в”Ђв”Ђ AI step N в”Ђв”Ђ` sono UX intenzionale, non leak architetturale), il flag "audio feedback" sta in settings perchГ© dove altro starebbe, CodeMirror conosce la convenzione `;; >>> AI` perchГ© la sostituzione in-place del blocco ГЁ la UX scelta consapevolmente. Г€ isolato in senso pulito: tre livelli di responsabilitГ  (core/describe per le features, prompt_store/history per infrastruttura cross-feature, capture/settings per servizi esterni adiacenti) e dipendenze lette in una sola direzione (AI dipende da capture, settings, codemirror; il viceversa non ГЁ vero). L'unico vero debito ГЁ la **tripla duplicazione del dispatch provider** (core, describe, auto_session): aggiungere un nuovo provider o cambiarne uno richiede di toccare tre file con strutture quasi identiche. Un'astrazione `ai.provider/call` con multimethod risolverebbe il problema in un giorno e spezzerebbe la convergenza accidentale che oggi tiene insieme i tre dispatch.

## 11.5 Voce

### 11.5.1 Pipeline

I sette file della pipeline si dividono in due livelli: stato/UI/contenuto da una parte, parsing/dispatch dall'altra.

**`voice/state.cljs`** (87 righe) custodisce un atom unico `voice-state` con sei sezioni ([L7-32](src/ridley/voice/state.cljs#L7-L32)): `:mode` (`:structure :turtle :help :ai`, default `:structure`), `:sub-mode` (`:selection :dictation` nil), `:language` (`:it :en`, default `:it`), `:voice` (listening, transcript, pending-speech, last-utterance), `:buffer`/`:cursor`/`:scene` (sync da `core.cljs` per dare al parser contesto editor), `:help` (results, page, query, view, highlight, replace-word). La separazione mode/sub-mode esiste perchГ© dictation e selection sono ortogonali ai mode principali. `:turtle` mode ГЁ dichiarato ma non implementato (parser ritorna nil in [`parser.cljs:145`](src/ridley/voice/parser.cljs#L145), commento "Phase 4").

**`voice/i18n.cljs`** (176 righe) ГЁ il dizionario delle frasi. `voice-commands` ([L4-132](src/ridley/voice/i18n.cljs#L4-L132)) ГЁ una mappa annidata `{:gruppo {:comando {:it [...] :en [...]}}}`. Sotto-gruppi: `:modes`, `:navigation`, `:editing`, `:structural`, `:insertion`, `:meta`, `:dictation`, `:selection`, `:units`, `:positions`, `:prepositions`, `:transforms`, `:help-browse`, `:language`. Plus `numbers` (cardinali IT/EN), `ordinals`, `fillers` (parole da strippare prima del match), `select-targets`, `negative-words` ("meno"/"minus" per literal negativi), `command-delimiters` (`#{"poi" "vai" "e poi" "quindi"}` IT, `#{"then" "go" "and then"}` EN вЂ” separatori in continuous mode). Г€ la cosa piГ№ vicina a un "engine grammaticale", tutto data driven.

**`voice/speech.cljs`** (269 righe) ГЁ puro browser-side. Nessun servizio cloud: usa `js/window.webkitSpeechRecognition || js/window.SpeechRecognition`, lang code `it-IT` o `en-US`. **Push-to-talk**: `start!` ([L184-208](src/ridley/voice/speech.cljs#L184-L208)) crea recognition con `continuous=false`, `interimResults=true`. `on-result-single` aggiorna il transcript parziale, `is-final` triggera la callback. `on-end-single` gestisce il caso in cui non si finalizza promuovendo il transcript parziale. **Continuous**: `start-continuous!` ([L210-237](src/ridley/voice/speech.cljs#L210-L237)) con `continuous=true`. `on-result-continuous` aggiorna parziale, ma se non finalizza dopo 1500ms auto-promuove il parziale come utterance via `partial-timer`. `on-end-continuous` riavvia il recognizer (browser termina sessioni dopo qualche minuto). `on-error-continuous` gestisce error: `no-speech`/`aborted` ignorati, `not-allowed` ferma definitivamente, altri errori contati e dopo 5 fail consecutivi stop.

**`voice/parser.cljs`** (148 righe) ГЁ il parser deterministico. **Lo ГЁ davvero**: tokenize в†’ strip fillers ("di", "per", "the", "of") в†’ match-phrases (longest first) contro tabelle i18n в†’ ritorna `{:action :keyword :params {...}}` o nil. Niente fuzzy matching, niente AI nel parsing. La priority order ([L120-148](src/ridley/voice/parser.cljs#L120-L148)) ГЁ: 1) mode-switch (sempre, "modo struttura"), 2) language-switch ("italiano"/"english"), 3) meta (undo/redo/run/stop), 4) dictation-enter (solo in struct mode), 5) mode-specific. Per mode `:ai` ritorna nil = "non ho capito, passa a LLM".

**`voice/modes/structure.cljs`** (229 righe) ГЁ il parser per il mode piГ№ ricco. Sub-parser sequenziali con `or`: navigation (next/prev/into/out/first/last/top con count opzionale), counted-navigation ("3 prossimo"), selection ("seleziona parola"), editing (delete/copy/cut/paste/change con valore preposition-aware), structural (slurp/barf/wrap con head opzionale/unwrap/raise/join), transform (keyword/symbol/hash/deref/cap/uppercase/number), insertion (insert/append/new-form/new-list/new-map con position e head opzionali). Tutto via `match-cmd` su tabelle i18n.

**`voice/help/`** ГЁ trattato in 11.5.2. **`voice/panel.cljs`** (97 righe) ГЁ solo UI status: HTML/CSS per il pannello in basso a destra che mostra mode, listening, transcript, pending. Quando mode=help include `voice/help/ui/render-html`.

**`voice/core.cljs`** (505 righe) ГЁ l'orchestratore. `init!` riceve la mappa di handlers da `core.cljs` (descritta nella correzione iniziale), `on-utterance` (push-to-talk) e `on-continuous-event` (continuous, con split su delimiter words "poi"/"vai") sono i due entry-point. Entrambi convergono su `process-command-text` che fa: se sub-mode `:dictation` insert literal (con check per parole di uscita "basta"/"stop"), altrimenti se mode=`:ai` e parser non match в†’ `handle-ai-input` в†’ `ai/generate`, altrimenti parse + execute-action. **`execute-action`** ([L36-275](src/ridley/voice/core.cljs#L36-L275)) ГЁ una mega-`case` su 30+ azioni che dispatcha ai callback (`insert`, `edit`, `navigate`, `execute`, `undo`, `speak`, `copy-fn`, `cut-fn`, `paste-fn`, `select-fn`) o a operazioni interne sullo stato voice (mode-switch, language-switch, help-search, help-browse, help-select).

I sei callback principali iniettati e dove li chiama `execute-action`. `:insert` ГЁ chiamato per `:insert`, `:append`, `:new-form`, `:new-list`, `:new-map`, `:dictation` text, e `:help-select` (insert template). `:edit` ГЁ chiamato per `:delete`, `:change`, `:slurp`, `:barf`, `:wrap`, `:unwrap`, `:raise`, `:join`, `:transform`, `:help-select` su replace-word. `:navigate` per `:navigate` e `:goto-line`. `:execute` solo per `:run` con `{:target :script}`. `:undo` per `:undo` e `:redo`. `:speak` (TTS opzionale) per feedback in mode-switch e in alcuni casi error. **Quale di questi ГЁ AI-mediato? Nessuno direttamente.** Tutti i sei callback fanno operazioni locali sull'editor. L'AI viene chiamata solo da `handle-ai-input`, che ГЁ raggiungibile solo quando `mode = :ai` AND il parser non ha matchato. TornerГІ sul punto in 11.5.3.

### 11.5.2 Help system come blob i18n

**Struttura confermata.** [`voice/help/db.cljs`](src/ridley/voice/help/db.cljs) ha due `def`: `tiers` (3 entries, `:ridley/:clojure-base/:clojure-extended` con `:order` e `:name {:it :en}`), e `help-entries` mappa-piatta `{keyword-symbol в†’ {:tier :category :template :doc {:it :en} :aliases {:it [...] :en [...]}}}`. Esempio da [L23-28](src/ridley/voice/help/db.cljs#L23-L28): `:f` (avanzamento turtle) con tier `:ridley`, category `:movement`, template `"(f 20)"`, doc IT "Muove avanti", EN "Move forward", aliases IT `["avanti"]`, EN `["forward"]`. Tre tier, 130 entries circa.

I tier sono **fasce di apprendimento, non prioritГ  di matching**. `tiers` ([L9-12](src/ridley/voice/help/db.cljs#L9-L12)) li nomina "Ridley DSL", "Clojure Base", "Clojure Esteso" con `:order` 1/2/3 per visualizzazione gerarchica. Lo `:order` ГЁ solo per ordinare la UI categories. Nel search engine ([`help/core.cljs/search-help`](src/ridley/voice/help/core.cljs#L49-L61)), il tier non entra nello score: tutti i match valgono uguali, sono ordinati solo per score di match testuale. L'idea ГЁ "principianti vedono tier 1 prima nella UI categories", ma una query come "circle" matcha entries di qualsiasi tier indistintamente.

**`:template` ГЁ una stringa esempio per inserimento.** Г€ il codice che viene effettivamente inserito quando l'utente seleziona quella voce dall'help: `(box 20)`, `(arc-h 20 90)`, `(register name (mesh))`. Supporta un marker `|` per indicare la posizione del cursore dopo l'insert ([`help/core.cljs/insert-template`](src/ridley/voice/help/core.cljs#L76-L87)): se presente, il template viene scisso in `text` (senza marker) e `cursor-offset` (dove posizionare il cursore). Non ГЁ grammatica per il riconoscimento vocale: il riconoscimento vocale ГЁ completamente separato e usa solo le tabelle in `voice/i18n.cljs`. Template e voice grammar non si parlano.

**`:aliases` ГЁ ricerca testuale, non grammatica vocale.** Sono frasi naturali in IT/EN per il fuzzy search del help. Quando l'utente ГЁ in mode `:help` e dice "cubo", il parser di help (`help/core.cljs/parse`) cattura "cubo" come `try-search` query, e `search-help` chiama `score-entry` che scorre `score-match` (sul nome simbolo) в†’ `score-alias-match` (sui alias del lang) в†’ `score-doc-match`. Score: 100 esatto, 80 prefix, 50 contains, alias gli stessi -5. Cosi "cubo" в†’ match alias di `:box` (score 95) в†’ entry mostrata. Esempi IT: `"avanti"` в†’ `:f`, `"cubo"`/`"scatola"` в†’ `:box`, `"sfera"` в†’ `:sphere`, `"cilindro"` в†’ `:cyl`, `"ruota"`/`"gira"` в†’ `:th`. Esempi EN: `"forward"` в†’ `:f`, `"cube"` в†’ `:box`, `"cylinder"` в†’ `:cyl`, `"turn"` в†’ `:th`. Gli alias **NON attivano il simbolo** nel sorgente: sono solo per trovarlo nell'help. Per attivarlo, l'utente dice il numero (1-7) della voce nella pagina visualizzata.

**`help/core.cljs`** (168 righe) ГЁ un fuzzy lookup engine **piГ№ voice command parser per il mode help**. Il search ranking ГЁ quello descritto sopra (3 tier di score con penalty per alias, max 70 risultati per query). `parse` ([L160-168](src/ridley/voice/help/core.cljs#L160-L168)) gestisce 5 azioni in prioritГ : `try-select` (numero 1-7 per selezionare voce nella pagina), `try-navigate` (next/prev pagina), `try-exit` (esci/exit/back), `try-browse` (tier names в†’ mostra tutte le voci di quel tier), `try-search` (fallback: tutto il resto ГЁ una query). Niente fuzzy oltre il ranking lineare, niente trigram o Levenshtein.

### 11.5.3 Il rapporto fra voce e AI

**Quando l'utente parla, NON passa mai per AI a meno che non sia esplicitamente in mode `:ai` AND il parser non matchi.** Il flusso ([`voice/core/process-command-text`](src/ridley/voice/core.cljs#L383-L414)):

1. Se `sub-mode = :dictation`: insert literal, no AI mai.
2. Se `mode = :ai`: prima `parser/parse-command` (per intercettare mode-switch, undo, language-switch, ecc.). Se match, dispatch normale. Se no match в†’ `handle-ai-input` в†’ `ai/generate`.
3. Altrimenti (`mode :structure :help :turtle`): `parser/parse-command`. Match в†’ dispatch. No match в†’ unrecognized, no AI.

Esempi concreti. **"fai un cubo"** in `:structure` mode (default): parser non ha pattern "fai un X". Tokenize в†’ `["fai" "un" "cubo"]` (filler "un" rimosso) в†’ `["fai" "cubo"]`. Nessun match in mode-switch, meta, structure parser. Risultato: `pending-speech "? fai cubo"`, niente succede. Per generare codice da quella frase, l'utente deve prima dire "modo ai" (mode-switch a `:ai`) e poi "fai un cubo" вЂ” solo allora `handle-ai-input` chiama `ai/generate`.

**"annulla l'ultimo comando"** in qualunque mode: parser fa try-meta, matcha `:undo` ("annulla"). Direct dispatch al callback `:undo`. No AI mai.

**"ruota la camera"**: in `:structure` mode, "ruota" non ГЁ in voice-commands (ГЁ un alias di `:th` nel help db, ma alias help non sono parole di comando). Match probabile: nessuno. In `:ai` mode, prima parser non matcha, poi handle-ai-input в†’ AI. Ma "ruota la camera" non ГЁ uno dei pattern del codegen (l'AI ГЁ per geometria, non per controllo viewport). Risultato probabile: l'AI risponde con clarification o codice non sensato.

**Il prompt usato per l'AI** quando handle-ai-input parte: `ai/generate` con tier-default (auto, normalmente tier-2 per modelli mid). System prompt ГЁ quello di `codegen/tier2` (o tier corrente) da `prompt_store`, che ГЁ di base una versione di `tier-2-prompt` di `ai/prompts.cljs`. User content ГЁ solo il transcript (no script-content corrente, no history вЂ” cioГЁ diverso da quello che fa la palette comando `/ai`). Quindi la voce passa una versione "leggera" del codegen, senza il contesto pieno del sorgente.

**Comandi puri senza AI**, lista degli intent supportati direttamente da `parser.cljs` per mode `:structure`:

- Mode-switch: `:structure :turtle :ai :help`.
- Language: `:it :en`.
- Meta: `:undo`, `:redo`, `:run`, `:stop`.
- Navigation: `:next :previous :into :out :first :last :top` con count.
- Goto-line.
- Editing: `:delete :copy :cut :paste :change` (con valore).
- Structural: `:slurp :barf :wrap` (con head opzionale) `:unwrap :raise :join`.
- Transform: `:keyword :symbol :hash :deref :capitalize :uppercase :number`.
- Insertion: `:insert :append :new-form :new-list :new-map` (con position e head).
- Selection: `:selection-enter :selection-exit :select-immediate`.
- Dictation: `:dictation-enter`.

E per mode `:help`: `:help-search :help-browse :help-select :help-next :help-prev :help-back :help-exit`.

In totale circa 35 azioni deterministic, contro 1 sola AI-mediated (`handle-ai-input` quando mode=`:ai` e parser non matcha).

### Sintesi 11.5

**La voce non sta sopra a tutto, ГЁ una shell deterministic con un mode AI esplicito come escape hatch.** Г€ un'interfaccia di accessibilitГ  con grammatica fissa (~35 azioni) che mappa direttamente a callback editor: insert, edit, navigate, execute, undo. L'AI ГЁ solo il fallback del mode `:ai`, raggiungibile solo cambiando mode esplicitamente. Il design ГЁ opposto a quello di assistant tipo Cursor o GitHub Copilot Voice: lГ¬ l'AI ГЁ il primary, il deterministic parser ГЁ l'eccezione. Qui il deterministic ГЁ il primary, l'AI ГЁ l'opt-in. La ragione ГЁ chiara dal contenuto del help db e delle tabelle i18n: il sistema ГЁ progettato per un utente blind che usa Ridley come strumento CAD vocale, non come traduttore natural-language в†’ codice. La precisione e la riproducibilitГ  del comando deterministic battono la flessibilitГ  dell'AI quando il microfono fa errori e l'utente non vede lo schermo.

## 11.3.3 Recupero: importatori SVG e STL

**`library/svg.cljs`** (299 righe) ha **due layer paralleli**, dichiarati nel docstring del modulo. Layer 1, **runtime function** (SCI binding): `svg`, `svg-shape`, `svg-shapes` esposte al DSL. Accettano stringhe SVG come input ([L67](src/ridley/library/svg.cljs#L67)): `(svg "...svg xml string...")`, internamente iniettano un container nascosto in `document.body` (`position: absolute, left: -9999px`, [L68-77](src/ridley/library/svg.cljs#L68-L77)) per usare il DOM SVG nativo del browser, e ritornano un handle opaque `{:type :svg-data :elements ... :viewBox ... :svg-root ... :dom-root}`. Tracciano i container in un atom per cleanup al SCI reset. `svg-shape` ([L139-183](src/ridley/library/svg.cljs#L139-L183)) estrae UNA shape da svg-data tramite `:id` o `:index`, samplata via `getPointAtLength` (browser-native, gestisce path/polygon/polyline/rect/circle/ellipse/line in `geometry-selector` [L31-33](src/ridley/library/svg.cljs#L31-L33)), applica il CTM cumulato (per gestire `<g transform=...>` annidati), flippa Y (SVG Y-down в†’ Ridley Y-up), scale, center, fix winding CCW. `svg-shapes` estrae tutte. Layer 2, **codegen** ([L273-289](src/ridley/library/svg.cljs#L273-L289)): `generate-library-source` parsa lo SVG via `discover-elements` (DOM transient solo per leggere id/tag, poi rimosso), embedda lo SVG come stringa escaped dentro il sorgente generato come `(def svg-data (svg "..."))`, e per ogni elemento crea `(def hexagon (svg-shape svg-data :id "hexagon"))` o `(def path-3 (svg-shape svg-data :index 3))` deduplicando i nomi. Supporta path/polygon/polyline/rect/circle/ellipse/line (sette tag SVG). Le curve di Bezier sono gestite implicitamente perchГ© `getPointAtLength` campiona la curva con `:segments` punti (default 64), quindi una curva diventa una poligonale con 64 segmenti. Multi-path: gestiti come elementi distinti, ognuno diventa un proprio `def`.

**`library/stl.cljs`** (257 righe) ha **lo stesso pattern di due layer**. Layer 1, **runtime SCI binding** `decode-mesh` ([L44-72](src/ridley/library/stl.cljs#L44-L72)): prende due stringhe base64 (vertici Float32Array packed e facce Uint32Array packed) e decodifica in `{:type :mesh :vertices [[x y z]] :faces [[i j k]] :creation-pose ...}`. Layer 2, **codegen** ([L234-252](src/ridley/library/stl.cljs#L234-L252)): all'import time parsa il file STL via `parse-binary-stl` o `parse-ascii-stl` (auto-detect via `binary-stl?`, [L78-87](src/ridley/library/stl.cljs#L78-L87) con check `byteLength == 84 + num_triangles * 50`), fa welding dei vertici (`vertex-map` con chiave stringa `"x,y,z"` per dedup), encoda in base64 i due array tipizzati. Il sorgente generato ГЁ semplice: `(def model-name (decode-mesh "vertices-b64..." "faces-b64..."))`. Plus un warning auto se la mesh ГЁ molto piccola (max-dim < 1.0): probabilmente ГЁ in metri invece di mm e suggerisce uno `(mesh-scale model 1000)`. Quindi il binding runtime ГЁ piГ№ semplice di SVG (no DOM, no parsing вЂ” solo decode), e il codegen ГЁ piГ№ pesante (tutto il parsing STL ГЁ fatto a import time, non a eval time).

**Il pannello UI** ([`library/panel.cljs:415-444`](src/ridley/library/panel.cljs#L415-L444)) ospita import via il bottone `+` (action button "Add" nell'header del panel) che apre un dropdown con tre voci: "New library" (modal prompt per nome), "Load SVG" (file picker con `accept=".svg"`), "Load STL" (file picker con `accept=".stl"`). Plus un secondo bottone `в†‘` separato ([L417-445](src/ridley/library/panel.cljs#L417-L445)) per "Import .clj library" (file picker `.clj,.cljs`). Drag&drop NON ГЁ implementato per import, solo per riordinare le librerie attive (vedi `setup-drag!` [L200-233](src/ridley/library/panel.cljs#L200-L233)). Filtro per estensione c'ГЁ ed ГЁ esplicito su ogni picker. Dopo import: la libreria ГЁ automaticamente attivata e (per SVG) entra in edit mode cosГ¬ l'utente vede subito il sorgente generato; per STL salta l'edit mode e chiama solo `on-change` (probabilmente perchГ© STL fa bundle binari non sensati da editare a mano).

**Risultato di entrambi gli importatori: una libreria a tutti gli effetti.** `storage/save-library!` viene chiamato con il source generato ([panel.cljs:375 e :405](src/ridley/library/panel.cljs#L375)) che produce un file `.clj` con header `;; Ridley Library: <nome>` (filesystem) o un JSON `ridley:lib:<nome>` (localStorage), entra in `_index.json`, ГЁ attivabile/disattivabile dal pannello come ogni altra libreria. Non c'ГЁ zona di storage separata. Una libreria SVG e una libreria STL sono indistinguibili da una libreria scritta a mano dall'utente, eccetto che la prima contiene `(svg "...big string...")` e `(svg-shape ...)`, e la seconda contiene `(decode-mesh "base64..." "base64...")`. Il risultato ГЁ elegante: tutto ГЁ codice, persistito come codice, evaluato come codice.

## 11.3 sidenote: edit mode del pannello librerie

**Cosa succede al click su "Edit".** [`enter-edit-mode!`](src/ridley/library/panel.cljs#L245-L256): legge il contenuto attuale dell'editor via callback `:get-editor-content`, lo salva in `panel-state` come `:saved-editor`, scrive il nome della libreria in `:editing`, carica il sorgente della libreria nel CodeMirror via callback `:set-editor-content`, e ri-renderizza il panel (che ora mostra una "edit bar" con `в—Ђ Back`, `Editing: <name>`, e `Save`, [L452-461](src/ridley/library/panel.cljs#L452-L461)). Plus una requires editor row in alto per aggiungere/togliere `:requires` con tag e `+ require` button ([L462-506](src/ridley/library/panel.cljs#L462-L506)).

**Cmd+Enter durante l'edit di una libreria.** Il CodeMirror ГЁ lo stesso, quindi Cmd+Enter triggera `evaluate-definitions` come al solito. Il sorgente che viene rivalutato ГЁ il sorgente della libreria (perchГ© ГЁ quello mostrato nell'editor in quel momento). Ma **la libreria stessa non ГЁ registrata come "active library" in modo speciale durante l'edit**: `make-sci-ctx` ([`repl.cljs:30-38`](src/ridley/editor/repl.cljs#L30-L38)) carica le librerie attive dal storage al momento dell'eval, non dal contenuto dell'editor. Quindi se la libreria che stai editando NON ГЁ ancora salvata, le tue modifiche non sono caricate come libreria fino al Save. PerГІ il contenuto dell'editor stesso ГЁ eseguito come "main script" вЂ” quindi i `(def ...)` nel sorgente diventano def globali nell'eval principale, non def nel namespace della libreria. Г€ un comportamento ambiguo: durante l'edit di `shapes`, Cmd+Enter valuta `(def hexagon ...)` come hexagon globale, non come `shapes/hexagon`. Ho l'impressione che sia un edge case non particolarmente curato, piГ№ che un design intenzionale.

**Uscita dall'edit mode.** Due strade. **Save** ([`save-and-exit-edit!`](src/ridley/library/panel.cljs#L258-L270)): legge il contenuto editor, lo salva via `storage/save-library!` (tenendo le `:requires` correnti), ripristina `:saved-editor` nel CodeMirror, resetta lo stato del panel, chiama `on-change` callback. **Back / Discard** ([`discard-and-exit-edit!`](src/ridley/library/panel.cljs#L272-L278)): ripristina e basta, niente save. Entrambi finiscono con il CodeMirror tornato allo stato pre-edit. Niente conferma "unsaved changes" su Back: clicca Back e perdi le modifiche.

**Workaround o design intenzionale?** Г€ **una via di mezzo, piГ№ vicino al workaround pragmatico**. Il design intenzionale sarebbe avere un secondo CodeMirror dedicato all'edit di librerie, magari come tab o split-pane. Quello che ГЁ implementato ГЁ "riusiamo l'editor principale, salviamo e ripristiniamo". I segnali di workaround sono tre: 1) il `:saved-editor` dentro `panel-state` ([L25](src/ridley/library/panel.cljs#L25)) tradisce l'intento di "sostituzione temporanea"; 2) Cmd+Enter durante edit ha il comportamento ambiguo descritto sopra; 3) Back senza conferma significa che le modifiche perse sono inevitabili ogni volta che l'utente sbaglia bottone. Architetturalmente la scelta ГЁ ragionevole per il volume di codice: un secondo editor sarebbe ~500 righe di setup CodeMirror duplicato, callback duplicate, sync state duplicate, e gli utenti probabilmente passano poco tempo in edit mode (rispetto al main script). Ma ГЁ un compromesso, non una soluzione clean.

## Tre cose ancora

**1. Auto-session ha tre dispatch provider stratificati, e mostra lo schema reale del cluster AI.** Per un round di `/ai-auto`: round 1 chiama `ai/core/generate` (dispatch in core.cljs:355), che internamente chiama uno dei `call-anthropic`/`call-openai-compatible`/`call-ollama`/`call-google` di core. Quindi una singola sessione `/ai-auto` di 5 round attraversa tutti e tre i dispatch hardcoded. Vale la pena nominarlo nel capitolo come esempio concreto del costo della duplicazione.

**2. Voice/AI ha un quirk silenzioso: il flusso AI dalla voce non manda lo script come contesto.** `handle-ai-input` ([voice/core.cljs:291-322](src/ridley/voice/core.cljs#L291-L322)) chiama `ai/generate transcript {:script-content (get-script)}` passando lo script. **3.
