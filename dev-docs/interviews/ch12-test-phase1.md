## 3. Cosa è testabile e cosa è strutturalmente difficile

**Testabile facilmente come è oggi.** Pure functions del motore turtle (movement, arc/bezier, joint-mode), già testate; pure functions del compilatore SDF ([src/ridley/sdf/core.cljs:200-240](src/ridley/sdf/core.cljs#L200-L240)) sarebbero testabili senza il server Rust perché `sdf-formula` produce un albero di tag; il parser voce ([src/ridley/voice/parser.cljs](src/ridley/voice/parser.cljs)); `validate-dependencies`/`detect-circular-deps` della library; `ridley.math` (62 righe pure); funzioni di easing in `anim/easing.cljs`; lo storage layer in `library/storage.cljs` per la parte di parsing/serialization; le pure functions di mesh (mesh-utils: simplify/laplacian/diagnose/merge-vertices). Anche il parser `extract-def-names` in [src/ridley/library/core.cljs:13](src/ridley/library/core.cljs#L13).

**Testabile ma serve infrastruttura non ancora presente.** Tutto ciò che dipende da Manifold WASM richiede un browser/headless Chrome o un build Node con il modulo WASM caricato (oggi i quattro test in `boolean_test.cljs` si auto-skippano, [test/ridley/manifold/boolean_test.cljs:90-91](test/ridley/manifold/boolean_test.cljs#L90-L91)). Lo stesso per il geo-server Rust: serve avviare il binario Tauri o un harness HTTP Rust dedicato. L'export STL/3MF è puro ma genera blob: serve un test che confronti byte-stream o conti record. Snapshot di mesh: serve una rappresentazione canonica (oggi mancano helper per "mesh =? mesh con ε topologica"); `test_helpers/watertight?` è un primo passo ma non basta. Golden image del viewport: richiede headless rendering Three.js, oggi non configurato. Mock LLM: serve un fake provider che restituisca risposte canned; `ridley.ai.core` non ha un'astrazione iniettabile, oggi va a fetch diretto. Sessione WebRTC: PeerJS richiede un signaling server simulato; i due ruoli (host/client) si possono testare in process con un fake transport, ma servirebbe rifattorizzare `peer.cljs` per estrarre l'I/O.

**Difficile da testare per natura del problema.** Voce in tempo reale (Web Speech API non esiste in Node, e il parsing dipende dalla qualità del riconoscimento upstream). WebXR e pose-tracking del visore. Drag-and-drop CodeMirror e binding di tastiera. Pilot mode ([src/ridley/editor/pilot_mode.cljs](src/ridley/editor/pilot_mode.cljs)) è interaction-driven. Camera capture ([src/ridley/viewport/capture.cljs](src/ridley/viewport/capture.cljs)) richiede WebGL context. Le animazioni dipendono da `requestAnimationFrame`; il bus collisioni in [src/ridley/anim/core.cljs:50](src/ridley/anim/core.cljs#L50) è event-loop bound.

## 6. Tre cose ancora

- Il file `src/ridley/editor/test_mode.cljs` ([src/ridley/editor/test_mode.cljs:1-5](src/ridley/editor/test_mode.cljs#L1-L5)) si chiama "test mode" ma è il backend del macro `tweak` (slider interattivi sui literal numerici), non infrastruttura di test. Il nome collide e può confondere chi cerca i test reali; il documento di spec corrispondente è [dev-docs/test-tweak-spec.md](dev-docs/test-tweak-spec.md), che a sua volta riporta nel titolo la parola "test".
- Il build target `:core` in [shadow-cljs.edn:30-58](shadow-cljs.edn#L30-L58) esiste con lo scopo dichiarato di "Validates browser-independence for future Ridley Desktop", ma non è collegato a nessuno script npm né alla CI: oggi è codice morto dal punto di vista pipeline, anche se il commento implica fosse pensato come gate di portabilità prima del passaggio Tauri. Visto che il JVM sidecar è stato rimosso, vale la pena valutare se reinserirlo come check o se eliminarlo.
- Lo `sci_harness.cljs` mocka registry/viewport/animazioni con `(fn [& _] nil)` ([test/ridley/editor/sci_harness.cljs:222-270](test/ridley/editor/sci_harness.cljs#L222-L270)), il che significa che `repl_test.cljs` verifica il comportamento dei macro DSL ma non verifica mai che `register-mesh!` venga effettivamente chiamato con i giusti argomenti. Le verifiche sui side effect del registry sono assenti perché lo stub è muto: il test passa anche se il macro decidesse di non registrare nulla.

## 7. Storia: test E2E perduti con il sidecar JVM

Ricostruzione dalla storia git. Il flusso ricordato a memoria ("scritti per la versione web, portati al JVM, poi cancellati") è giusto come direzione ma con una correzione: i test web non sono stati spostati, sono stati duplicati sul JVM, e poi è stata cancellata la copia JVM. La suite originale lato browser è ancora in repo.

**Fase 1: nascita lato web (gennaio-marzo 2026).** Il primo test cljs in assoluto è il commit `c08ccf3` del 24 gennaio 2026, su movimenti turtle. Il 6 febbraio arrivano in burst i commit "Sprint 1/2" (`bf65ab8`, `8a25206`, `d0fda14`, `0f88e4c`) con `internals_test`, `extrusion_test`, `arc_bezier_test`, `geometry/operations_test`, `scene/registry_test`. Il 7 febbraio (`39e62e2`, "Sprint 4") nasce il workflow CI insieme a `modules_test`. Sempre il 7 febbraio (`23fa647`) nasce [test/ridley/editor/repl_test.cljs](test/ridley/editor/repl_test.cljs) a 449 righe, accompagnato da [test/ridley/editor/sci_harness.cljs](test/ridley/editor/sci_harness.cljs) a 718 righe: è la suite E2E del DSL via SCI lato browser. Il 22 febbraio (`2855a9d`) cresce a ~775 righe assorbendo i turtle scopes. Il 4 marzo (`9942168`) viene aggiunto `shell_test`.

**Fase 2: port sul JVM (3-15 aprile 2026).** Il 3 aprile il commit `b631d7b` introduce `desktop/ridley-jvm/test/ridley/jvm/eval_test.clj` a 543 righe, 64 test e 78 asserzioni. Il messaggio dichiara letteralmente "Port missing DSL bindings from CLJS to JVM eval engine" e "Add test infrastructure (cognitect test-runner) and comprehensive test suite". I nomi dei deftest si sovrappongono fortemente con `repl_test.cljs` (`bloft-*`, `loft-n-*`, `turtle-scope-*`, `revolve-*`, `attach-face-*`, `extrude-closed`) ma con qualche rebrand cosmetico (`extrude-basic` invece di `extrude-inline`). Il 3-4 aprile, in nove commit consecutivi (`cf4d644`, `12c6a3f`, `a43606f`, `a5c8872`, `7bdc5e3`, `e06ffca`, `034d5bd`, `4c35a3e`, `d69fe8b`, `16baf6d`), il file cresce a 1394 righe assorbendo registry, faces, text/SVG, chamfer/fillet, JTS booleans, warp, scene registry, query face selection. Il 15 aprile (`70e0919`) si aggiungono SDF formula e TPMS surfaces, portando il file a 1594 righe e ~80+ deftest.

**Fase 3: rimozione (23 aprile 2026).** Il commit `f5e3fdd` cancella tutto `desktop/ridley-jvm/`, 33 file, e nel messaggio elenca "tests" fra le cose rimosse: "Delete desktop/ridley-jvm/ (33 files — eval engine, server, geometry, turtle, SDF, manifold, tests)".

**Cosa è sopravvissuto, cosa no.** L'originale [test/ridley/editor/repl_test.cljs](test/ridley/editor/repl_test.cljs) è ancora in repo, ultima modifica 22 febbraio. Esercita il DSL via SCI in Node con tutti i side effect mockati. Quello che è scomparso è la copia JVM, che era una superset. I sottosistemi che la copia JVM copriva e che oggi non hanno più copertura E2E sono: SDF tree construction (sphere/box/blend/morph/shell-offset/move/rotate/scale/formula), 2D booleans JTS, chamfer e fillet, query-based face selection (`find-faces`, `face-at`, `face-nearest`), registry di path e shape, text shapes con triangolazione JTS, deformazione spaziale `warp`. Per questi cluster la perdita di test è netta: ne resta solo lo script DSL manuale [examples/sdf-sci-test.clj](examples/sdf-sci-test.clj) come smoke test per l'SDF (ma richiede il server Rust acceso, vedi sezione 1).

**Stato della superset perduta.** L'ultima versione vivente del file (commit `f5e3fdd^` = `70e0919`) può essere recuperata con `git show 70e0919:desktop/ridley-jvm/test/ridley/jvm/eval_test.clj`. È Clojure JVM e usa `clojure.test` direttamente contro `ridley.jvm.eval/eval-script`, quindi non è eseguibile come tale dalla suite cljs corrente: per riportarla a vita servirebbe riscriverla contro il contesto SCI di `sci_harness.cljs` (oppure aggiungere bindings JTS/SDF al harness).
