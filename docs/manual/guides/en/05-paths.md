# Paths: recording movement

## What a path is

In chapter 4 we wrote turtle commands directly inside `extrude`: `(extrude (circle 10) (f 20) (th 45) (f 20))`. The path is defined on the spot and used immediately. But sometimes you need the same path in several places: as the trajectory of an extrusion, as the position of components along a skeleton, as a curve to display in the viewport. Rewriting it every time is fragile and tedious.

`path` records a sequence of turtle commands as reusable data, without executing them on the main turtle.

<!-- example-source: path-basic -->
```clojure
(def curl-path
  (path
    (dotimes [i 12]
      (f 20) (th (if (odd? i) 40 10)))))

(register tube-a (extrude (circle 5) curl-path))
(u -50)
(register tube-b (extrude (rect 8 4) curl-path))

```

`path` captures the commands inside its body and returns a piece of data. `def` gives it a name. Here the loop runs twelve times, and at each turn `if` picks the angle: a wider 40° on the odd steps, a gentler 10° on the even ones, so the path curls instead of running straight. The same `curl-path` is then used in two different extrusions, with two different shapes, in two different positions. The path is written only once.

Inside `path` you can use any Clojure code: `dotimes` for loops, `if` for choices, `let` for local variables: the example above uses the first two. The Clojure code runs at the moment the path is created, but only the turtle commands that code produces end up in the path. The `(dotimes [i 12] (f 20) (th ...))` runs during the recording: the `dotimes` loops twelve times, and what ends up in the path is 12 `f` and 12 `th`, not a loop, and not the `if` either: only the angle it chose at each step. The path does not know those commands were generated by a loop.

A path does not change the main turtle. When you write `(def p (path (f 100)))`, the turtle stays where it was. The path is only data, a recorded sequence of instructions. They become real movement only when a consumer interprets them: `extrude`, `loft`, `follow-path`, `text-on-path`.

## Visualizing a path

A path is invisible data: it records commands, it does not produce geometry. To see it in the viewport you use `follow-path`.

`follow-path` runs the path's commands on the current turtle. If the pen is down (the default), the turtle traces the path's lines in the viewport.

<!-- example-source: follow-path-basic -->
```clojure
(def my-path (path (f 20) (arc-h 15 90) (f 20)))
(follow-path my-path)
```

The turtle travels the path and draws the trace as a construction line. Be careful: `follow-path` really does move the turtle. After the call, the turtle is at the end of the path. If you do not want it to move, wrap it in a `turtle` scope:

```clojure
(turtle (follow-path my-path))
;; the outer turtle has not moved
```

`follow-path` is also useful as a compositional tool: a path defined elsewhere can be "replayed" on the turtle to position it at a precise point before building something.

The lines traced by the turtle (whether via `follow-path` or from direct movements) can be shown or hidden with the "Lines" button in the viewport toolbar, or with `(show-lines)` / `(hide-lines)` from code.

`show-lines` and `hide-lines` control the global visibility of the lines already traced. If instead you want a single movement to leave no trace, it is the turtle's pen that decides: `(pen :off)` (or `pen-up`) lifts it, and the following movements do not draw; `(pen :on)` (or `pen-down`) lowers it again. The pen is a turtle state, inherited by nested `turtle` scopes.

## Marks: tags inside a path

A path is not just a trajectory: it is also a place to leave markers. `mark` records the turtle's pose (position, direction, orientation) at a point along the path, with a name.

<!-- example-source: mark-basic -->
```clojure
(def arm-path
  (path
    (f 30)
    (mark :elbow)
    (th 45)
    (f 20)
    (mark :hand)))

(register arm (extrude (circle 3) arm-path))

(register joint
  (attach (sphere 5)
    (move-to arm-path :at :elbow :align)))

(register tip
  (attach (cone 0 4 8)
    (move-to arm-path :at :hand :align)))
```

Two marks: `:elbow` at 30 forward (before the curve), `:hand` at the end of the path (after the curve). The marks produce no geometry and do not affect the path: they are just labels attached to a pose.

`move-to` is available inside `attach` and `attach!`: it brings the turtle to the pose of the given mark, positioning the mesh that is being attached. `:align` aligns the direction too, not just the position. So the sphere ends up exactly at the elbow, and the cone exactly at the hand, oriented in the direction of the path at that point.

This is the fundamental skeleton pattern: a path with marks defines *where* things are, the shapes and primitives define *what* things are, and `move-to` connects them. Chapter 8 (Assembly) builds whole assemblies on this principle. Here we care about the basic mechanism.

`anchors` returns the complete map of all the marks of a path:

```clojure
(anchors arm-path)
;; => {:elbow {:pos [...] :heading [...] :up [...]}
;;     :hand  {:pos [...] :heading [...] :up [...]}}
```

To query a single mark without going through `move-to`:

```clojure
(mark-pos arm-path :elbow)   ; => [30 0 0]
(mark-x arm-path :elbow)     ; => 30
(mark-z arm-path :elbow)     ; => 0
```

`mark-pos` returns the 3D coordinates of the mark. `mark-x`, `mark-y`, `mark-z` return the individual components. They are useful when you need the numeric position, not the movement of the turtle.

While `mark` leaves a marker during the recording of the path, `add-mark` adds one to an already-built path, at a fractional position along its length: `(add-mark p :apex 0.5)` puts `:apex` at the path's midpoint, measured by arc length. It is useful when the point you need does not coincide with any command of the path, for example the apex of a curve. We will see it at work in §11.3.

## Side-trip: branches that return to the spine

Often a path has a main spine (the axis along which the structure develops) and side branches where marks are left for accessory components. Without a dedicated mechanism, each branch would have to manually undo its own movements to bring the turtle back to the spine: turn right, forward 20, leave a mark, back 20, turn left. Fragile and tedious.

`side-trip` solves the problem: it runs its body as a sub-path, records everything (marks included), and then automatically returns the turtle to the position it was in before entering.

<!-- example-source: side-trip-basic -->
```clojure
(def skel
  (path
    (mark :start)
    (f 40)
    (side-trip
      (th 60)
      (mark :start-branch)
      (f 27)
      (mark :end-branch))
    (mark :after)
    (f 20)
    (mark :end)))

(register tree
  (mesh-union
    (extrude (circle 5) skel)
    (attach (cyl 10 2) (move-to skel :at :start))
    (attach (cyl 12 2) (move-to skel :at :after))
    (attach (cyl 15 2) (move-to skel :at :end))
    (turtle (:start-branch skel)
      (extrude (circle 2) (f 27)))))
```

The spine runs from `:start` to `:end` passing through `:after`. The `side-trip` breaks off from the spine at 40 forward, turns by 60°, and leaves two marks (`:start-branch` and `:end-branch`). Then the turtle automatically returns to where it was before the `side-trip` (at 40 on the spine), and the path continues with `:after` and `:end`.

Three disks placed with `move-to` mark the points on the spine. The side branch is built with a different syntax: `(turtle (:start-branch skel) ...)`. A path behaves like a map of its anchors, so `(:start-branch skel)` returns the pose of the mark `:start-branch`, and `turtle` opens a scope positioned at that pose. Inside the scope, `(extrude (circle 2) (f 27))` builds the branch.

This syntax is the most direct way to build geometry starting from a mark. The difference from `move-to` (which works only inside `attach`): `turtle` with an anchor opens a whole scope in which you can do several operations, not just position one mesh.

`turtle` accepts anchors in three equivalent forms:

```clojure
(turtle (:anchor-name my-path) ...)    ; implicit accessor (the most common)
(turtle my-path :at :anchor-name ...)  ; :at sugar
(turtle :pose some-pose ...)           ; explicit pose from any source
```

The first form is the most compact and the one you will use almost always. The second makes it explicit that you are navigating to an anchor of a path. The third accepts any pose (not only from a path), useful when the pose comes from a computation or from `(:end-face result)`.

### Pattern: a helper with side-trip

Another frequent pattern: wrapping the side-trip in a helper function that parametrizes the direction and orientation of the branch.

<!-- example-source: side-trip-helper -->
```clojure
(defn arm [side rot mname]
  (path
    (side-trip
      (th (if (pos? side) 90 -90))
      (tr rot)
      (f (abs side))
      (mark mname))))

(def skel
  (path
    (f 50)
    (follow (arm  27 15 :a-1))
    (follow (arm -27 30 :a-2))
    (f 30)
    (follow (arm  27 45 :a-3))
    (follow (arm -27 60 :a-4))))

(register stuff
  (mesh-union
    (extrude (circle 3) skel)
    (concat-meshes (for [k (keys (anchors skel))]
      (attach (box 10 5 2) (move-to skel :at k :align))))))
```

`arm` creates a sub-path that breaks off from the spine, goes sideways by `side` (positive = right, negative = left), rotates with `tr` to vary the orientation, and leaves a mark. `follow` grafts it into the spine. Four calls with different parameters place four marks along the spine, each at a different angle.

The final part iterates over all the marks with `for` and places a box on each, aligned to the mark's pose. `concat-meshes` collects the meshes produced by the `for` into a list that `mesh-union` can merge with the extruded tube.

The `side-trip` nests without trouble: a side-trip can contain a `follow` that contains another side-trip. Each level saves and restores its own pose.

## Paths from coordinates

So far we have built paths with turtle commands: forward, turn, arc. But sometimes the path is defined as a list of points (coordinates measured from a drawing, imported points, computed waypoints). Two constructors create paths directly from coordinates.

### poly-path

`poly-path` creates a path from pairs of coordinates, like `poly` for shapes.

<!-- example-source: poly-path-basic -->
```clojure
(def profile-path (poly-path 0 0  20 0  20 30  10 40  0 30))
(register rail (extrude (circle 3) profile-path))
```

Five points, an open path that connects them with straight segments. The coordinates are X/Y pairs in the turtle's plane. `poly-path` also accepts a vector: `(poly-path [0 0 20 0 20 30 10 40 0 30])`.

`poly-path-closed` does the same thing but closes the path by returning to the first point:

<!-- example-source: poly-path-closed -->
```clojure
(def frame-path (poly-path-closed 0 0  40 0  40 30  0 30))
(register frame (extrude-closed (circle 2) frame-path))
```

A closed rectangle, useful as a path for `extrude-closed`.

### quick-path

`quick-path` (alias `qp`) is the compact constructor we already saw in chapter 4: numbers and angles alternate.

<!-- example-source: quick-path-zigzag -->
```clojure
(def zigzag (qp 20 60 20 -120 20 60 20))
(register bar (extrude (rect 5 3) zigzag))
```

`qp` is handy for straight paths with known angles. `poly-path` is handy when you have the coordinates. `path` with turtle commands is the most flexible: arcs, beziers, side-trips, marks.

## Paths as shape embryos

A path describes a route. A shape describes an outline. But the two concepts touch: a closed path in the plane is also an outline, and an outline with a thickness is also a solid. Two functions cross the boundary.

### path-to-shape

`path-to-shape` converts a path into a 2D shape, projecting the waypoints onto the right-up plane.

<!-- example-source: path-to-shape-basic -->
```clojure
(def rim
  (path
    (mark :foot-1) (f 30) (th 60)
    (mark :foot-2) (f 30) (th 60)
    (mark :foot-3) (f 30) (th 60)
    (mark :foot-4) (f 30) (th 60)
    (mark :foot-5) (f 30) (th 60)
    (mark :foot-6) (f 30)))

(register plate (extrude (path-to-shape rim) (f 4)))
```

The same path `rim` serves two purposes: as a shape (via `path-to-shape`) to extrude the plate, and as a skeleton (via the marks) to place the feet. A single piece of data, two consumers, no chance they diverge.

This is the main use case of `path-to-shape`: when the profile and the skeleton are the same curve, there is no need to describe them twice.

There is more. When the path you convert contains `mark`s, `path-to-shape` does not discard them: it records them on the shape as references to the corresponding point index. From there the marks travel with the shape, and extruding, lofting or revolving that shape turns them into anchors of the resulting mesh, reachable with `(move-to mesh :at :name)` like any skeleton anchor.

<!-- example-source: path-to-shape-marks -->
```clojure
(def prof
  (path (f 20) (th 90) (f 20) (th 90) (mark :corner)  (f 20) (th 90) (f 20)))

(register block (extrude (path-to-shape prof) (f 15)))

(register pin
  (attach (cone 2 0 5) (move-to :block :at :corner :on 1)))
```

The profile is a square with a mark on one corner. The extruded box inherits that mark as an anchor, and `move-to :block :at :corner :on 1` attaches a pin there without recomputing where the corner is: the mesh carries its notable points with it, with no need to keep the original path around.

Since a mark is a point index and not a fixed coordinate, it follows the shape even when the shape is deformed. A shape-fn like `tapered` or `twisted` scales and rotates the profile's points along the loft, and the mark stays on its corner as the profile changes. This holds as long as the shape-fn preserves the number and order of the points: one that resamples the profile, adding points, shifts the indices and the marks fall back to their base positions. `(slice-mesh mesh :on t)` relies on this property, returning the generative profile at a given height of the loft with the marks in the right place (chapter 7.5).

A current limitation: `path-to-shape` works only on paths confined to the right-up plane (commands `f` and `th`). If the path contains `tv` or `tr` (movements out of the plane), they are silently ignored. A true 3D→2D projection is in the Roadmap.

### stroke-shape

`stroke-shape` takes a path and gives it a thickness, producing a 2D shape with an outline that follows the path at a fixed distance.

<!-- example-source: stroke-shape-basic -->
```clojure
(def curve (path (f 20) (arc-h 10 90) (f 15)))
(register ribbon (extrude (stroke-shape curve 3) (f 2)))
```

The path `curve` becomes a shape 3 wide (1.5 per side) that follows the route, and extruded it produces a flat ribbon. It is like drawing the path with a marker of fixed width and then extruding the stroke.

`stroke-shape` accepts options for the caps and the joins:

```clojure
(stroke-shape curve 3
  :start-cap :round    ; rounded start cap
  :end-cap :flat       ; flat end cap
  :join :miter)        ; mitered joins
```

The available caps are `:round` and `:flat`. The joins are `:round` (default), `:miter` (pointed), `:square`.

## Things that accept a path

Paths are data. They become useful when something consumes them. Here is the complete map of the consumers.

### extrude, loft, extrude-closed

The main consumer. They drag a shape along the path's route. We saw them throughout chapter 4.

<!-- example-source: where-extrude-loft -->
```clojure
(def my-path (path (f 20) (arc-h 15 90) (f 20)))
(register tube (extrude (circle 5) my-path))
(u 30)
(register vase (loft (tapered (circle 15) :to 0.5) my-path))
```

### follow-path

Runs the path's commands on the current turtle, moving it and tracing the lines. We saw it in 5.2.

### text-on-path

Lays 3D text along the path's curve. Each letter is positioned and oriented following the route.

<!-- example-source: where-text-on-path -->
```clojure
(def curve (path (dotimes [_ 40] (f 2) (th 3))))
(register title (text-on-path "Hello Ridley" curve :size 15 :depth 3))
```

Chapter 13 (Text) covers `text-on-path` in detail.

### move-to (inside attach)

Brings the turtle to the pose of a mark inside the path. Available only in the body of `attach` and `attach!`. We saw it in 5.3.

```clojure
(attach (sphere 5) (move-to my-path :at :elbow :align))
```

### with-path, path-to, turtle

`with-path` opens a scope in which the marks of a path become navigable anchors. Inside the scope, `turtle` with the name of an anchor positions the turtle on that mark, and `path-to` builds a path from the current position to a target anchor.

<!-- example-source: where-with-path -->
```clojure
(def skel (path (mark :base) (f 15) (mark :mid) (f 12) (mark :top)))

(with-path skel
  (register lower
    (turtle :base (extrude (circle 2.5) (path-to :mid)))))

(with-path skel
  (register upper
    (turtle :mid (extrude (circle 1.2) (path-to :top)))))
```

`with-path` is the main pattern for building several pieces along the same skeleton. Each piece is extruded from one mark to the next, and the pieces are automatically aligned because they share the same skeleton. Chapter 8 (Assembly) uses this pattern extensively.

### turtle with an anchor

Opens a scope positioned on the pose of a mark. Unlike `move-to` (which works only inside `attach`), `turtle` lets you run several operations from the mark's pose: extrusions, registrations, sub-assemblies. We saw it in 5.4.

```clojure
(turtle (:elbow my-path)
  (register joint (sphere 5)))
```

### anchors

Returns the mark→pose map of a path. Used to iterate over all the marks, typically in a `for` that places components.

```clojure
(for [m (keys (anchors my-path))]
  (attach (cyl 3 10) (move-to my-path :at m :align)))
```

`get-anchor` is the single-name version: inside a `with-path` scope it reads the full pose (`:position`, `:heading`, `:up`) of an anchor from the current turtle, or `nil` if that name is not in scope. Useful for measurements and runtime computations, for example the distance between two marks.

### path-to-shape, stroke-shape

Convert a path into a 2D shape. We saw them in 5.7.

### bezier-as

Takes an existing path and traverses it as a smooth Bezier curve, smoothing the corners. It is used inside another `path` or inside `extrude`:

```clojure
(register smooth-tube
  (extrude (circle 4)
    (path (bezier-as my-angular-path))))
```

### fit

Scales a path to a target size on one or both axes:

```clojure
(def tall-path (fit my-path :y 180))
(register column (revolve (path-to-shape tall-path)))
```

### subpath-y

`subpath-y` cuts from a path the portion contained in a height band, measured as the distance from the start of the path, and re-anchors it so it starts from Y=0. It is useful above all for taking a section of a revolution profile without dragging the rest along.

<!-- example-source: subpath-y-band -->
```clojure
(def profile (path (f 5) (th 80) (f 15) (arc-h 5 -160) (f 15)))
(register slice (revolve (path-to-shape profile)))
(f 50)
(register slice2 (revolve (path-to-shape (subpath-y profile 6 18))))

```

Only the 6..18 band of the profile is revolved. It returns `nil` if the resulting band has fewer than two points.

The path is the most versatile piece of data in Ridley after the mesh: it describes a route, marks positions, and lends itself to different consumers without changing. The same path can be the trajectory of a tube, the skeleton of an assembly, and the profile of a plate, all in the same script.
