# decode-mesh

## Signature

`(decode-mesh vertices-b64 faces-b64)`

## Description

Rebuild a mesh from a packed binary representation: a base64-encoded
Float32 vertex array (`x0 y0 z0 x1 y1 z1 …`) and a base64-encoded
Uint32 face index array (`a0 b0 c0 a1 b1 c1 …`). Returns a mesh
value.

`decode-mesh` is the binding behind library-imported STL parts: the
importer stores each part as two base64 blobs in the generated code,
and the user code calls `decode-mesh` at eval time to reconstruct the
mesh. The result's creation-pose position is anchored at the
bounding-box center so subsequent in-place `translate` / `rotate`
operations pivot at the visual center.

Typical end-user code does NOT call `decode-mesh` directly — it
appears inside autogenerated library code. The binding is exposed in
SCI so that hand-written code can also construct mesh literals from
binary payloads (e.g. when embedding a fixed asset directly in a
sketch).

## Parameters

- `vertices-b64` — string. Base64-encoded little-endian Float32
  array of vertex coordinates.
- `faces-b64` — string. Base64-encoded little-endian Uint32 array
  of triangle indices into the vertex array.

## Notes

- The creation-pose is set to `[bbox-center 0 0 0]` with default
  heading/up so subsequent `translate`/`rotate`/`scale` pivots
  behave sensibly out of the box.
- Material/colour metadata is NOT embedded in the binary payload —
  it must be set via `material` / `color` after decoding.
- For STL import via the panel UI (rather than as embedded data),
  see the library system. `decode-mesh` is the low-level primitive
  underneath.
