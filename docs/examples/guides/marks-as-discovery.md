# Marks as discovery

> Generate marks programmatically with naming patterns, then recover them via `(anchors path)` and a filter.

## When to use this

Reach for this pattern when:

- The number of marks in a skeleton depends on a parameter (a list, a count). Hand-listing them is tedious; programmatic generation keeps the skeleton honest.
- You want to attach the same component to a *family* of marks — all the brackets, all the bolt holes, all the leg sockets — without enumerating each by name.
- The mapping from mark to component is uniform: one component per mark, with the mark's pose deciding placement and orientation.

If your skeleton has three or four hand-named marks (`:pin-axis`, `:slot-a`, `:slot-b`) and each one carries a *different* component, this pattern is overkill — just enumerate them. Use it when the marks form a regular family.

## How it works

The pattern has three steps.

**1. Generate marks programmatically.** Inside a `path`, build mark names from data — typically a prefix plus an index. The prefix groups marks by role; the index distinguishes them within a group.

```clojure
(def row-skel
  (path
   (mark :end-post-start)
   (f 20)
   (doseq [i (range 3)]
     (mark (keyword (str "mid-post-" i)))
     (f 20))
   (mark :end-post-finish)))
```

Five marks total, with two distinct roles: `:end-post-start` and `:end-post-finish` (the corners), and `:mid-post-0`, `:mid-post-1`, `:mid-post-2` (the middle posts).

**2. Recover them with `anchors`.** `(anchors path)` returns a map of `{mark-name → pose}`. `(keys …)` gives the list of names.

```clojure
(keys (anchors row-skel))
;; => (:end-post-start :mid-post-0 :mid-post-1 :mid-post-2 :end-post-finish)
```

**3. Filter and map by role.** Each role gets its own filter and its own component. The end posts are heavier; the mid posts are lighter:

```clojure
(def end-post (cyl 8 40))
(def mid-post (cyl 4 30))

(register fence
  (concat-meshes
   ;; End posts at the corners.
   (for [m (filter #(re-find #"^end-post-" (name %))
                   (keys (anchors row-skel)))]
     (attach end-post (move-to row-skel :at m :align)))
   ;; Mid posts in between.
   (for [m (filter #(re-find #"^mid-post-" (name %))
                   (keys (anchors row-skel)))]
     (attach mid-post (move-to row-skel :at m :align)))))
```

The filter is what turns `keys` into a *role-aware* selector. Without it, you'd have two ways forward: enumerate marks by name (which defeats programmatic generation), or build separate skeletons (which fragments the source of truth). The filter keeps one skeleton, two roles, two components.

## Example: bracket pairs in the hinge pilot

The pilot example (`assembly/01-hinge-c-profile.cljs`) uses this pattern to place bracket pairs along the hinge axis. The number of pairs is driven by a parameter, `bracket-axials`, listing the axial fractions where pairs should sit:

```clojure
(def bracket-axials [0.625 -0.33])
```

Two values, so two bracket pairs (four brackets total). Each pair is generated by a sub-path that drops two marks named `:bracket-1-l`, `:bracket-1-r`, `:bracket-2-l`, `:bracket-2-r`:

```clojure
(defn bracket-pair [i k]
  (path
   (side-trip
    (f (* k H))
    (follow (arm (/ outer-w 2) (/ outer-d 2)
                 (keyword (str "bracket-" (inc i) "-l"))))
    (follow (arm (- (/ outer-w 2)) (/ outer-d 2)
                 (keyword (str "bracket-" (inc i) "-r")))))))
```

At assembly time, all bracket marks are recovered in one filter and attached to a single bracket mesh:

```clojure
(def staffe
  (concat-meshes
   (for [m (filter #(re-find #"bracket" (name %))
                   (keys (anchors hinge-skel)))]
     (attach bracket (move-to hinge-skel :at m :align)))))
```

Adding a third pair is a one-character change: `[0.625 -0.33 0.1]`. The path generation, the filter, and the loop all adapt automatically.

## Common pitfalls

- **Naming patterns that catch unwanted marks.** A filter like `#"bracket"` will match any mark with "bracket" in its name. If the skeleton later gains a `:bracket-spec` or `:end-bracket` mark for a different purpose, it gets swept up by accident. Use distinctive prefixes and anchored regexps (`#"^bracket-"`) when in doubt.

- **Relying on filter order.** `(keys …)` returns marks in insertion order in modern ClojureScript, but filtering and mapping shouldn't *depend* on order if the pattern is genuinely about a set of equivalent marks. If order matters (e.g. left before right), encode it in the names (`-l`/`-r` suffix) and sort or partition explicitly.

- **Generating marks without a stable scheme.** Names built from random or non-deterministic data are hard to filter back. Stick to data already present in your code — indices, coordinates, named keys.
